# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

VibeVoice is Microsoft's open-source frontier voice AI framework containing:
- **VibeVoice-ASR**: Unified speech-to-text model for 60-minute long-form audio with speaker diarization and timestamps
- **VibeVoice-TTS**: Long-form multi-speaker text-to-speech (up to 90 minutes)
- **VibeVoice-Realtime**: Lightweight real-time streaming TTS (0.5B params, ~300ms first-audio latency)

Core innovation: Continuous speech tokenizers (Acoustic + Semantic) operating at 7.5 Hz frame rate with next-token diffusion generation.

## Build & Development Commands

```bash
# Install package in editable mode
pip install -e .

# Install with streaming TTS dependencies
pip install -e .[streamingtts]

# LoRA fine-tuning for ASR (requires additional dependencies)
cd finetuning-asr
pip install peft
torchrun --nproc_per_node=1 lora_finetune.py --model_path microsoft/VibeVoice-ASR --data_dir ./toy_dataset --output_dir ./output --num_train_epochs 3 --bf16
```

## Running Demos

```bash
# ASR Gradio demo
python demo/vibevoice_asr_gradio_demo.py --model_path microsoft/VibeVoice-ASR --share

# ASR inference from file
python demo/vibevoice_asr_inference_from_file.py --model_path microsoft/VibeVoice-ASR --audio_files /path/to/audio.wav

python demo/realtime_model_inference_from_file.py --model_path microsoft/VibeVoice-Realtime-0.5B
```

## Code Architecture

### Core Package Structure (`vibevoice/`)

- **`modular/`** - Model architectures and configurations
  - `modeling_vibevoice.py` - TTS model (LLM + speech connectors + diffusion head)
  - `modeling_vibevoice_asr.py` - ASR model (encoder + Qwen2 language model)
  - `configuration_vibevoice.py` - Config classes for all model types
  - `modular_vibevoice_tokenizer.py` - Speech tokenizers (acoustic/semantic encoding)
  - `modular_vibevoice_diffusion_head.py` - DPM diffusion head for TTS
  - `streamer.py` - AudioStreamer classes for streaming generation

- **`processor/`** - Input processing
  - `vibevoice_asr_processor.py` - ASR audio preprocessing and tokenization
  - `vibevoice_tokenizer_processor.py` - Audio feature extraction
  - `vibevoice_processor.py` - Main VibeVoice processor

- **`schedule/`** - Diffusion scheduling
  - `dpm_solver.py` - DPM-Solver for diffusion sampling
  - `timestep_sampler.py` - Training timestep sampling

### Integration (`vllm_plugin/`)

vLLM integration for accelerated inference. Plugin auto-registers on import via `vllm.general_plugins` entry point in `pyproject.toml`.

### Data Flow

**ASR Pipeline:**
```
Audio (24kHz) → AcousticTokenizer → AudioTokens → SpeechConnector → LLM Input
Audio (24kHz) → SemanticTokenizer → SemanticTokens → SpeechConnector → LLM Input
                                                                    ↓
                                    Structured JSON Transcription ← Generate
```

## Key Configuration Classes

- `VibeVoiceConfig` - Base config (LLM hidden_size, speech tokenizer configs, diffusion head)
- `VibeVoiceASRConfig` - ASR-specific (acoustic/semantic tokenizer configs, decoder config)
- `VibeVoiceStreamingConfig` - Streaming TTS config
- `VibeVoiceAcousticTokenizerConfig` / `VibeVoiceSemanticTokenizerConfig` - Speech tokenizer configs

## Development Guidelines (from CONTRIBUTING.md)

- **Code Minimalism**: Research-purposed code, not enterprise software. Avoid over-engineering, excessive abstraction, or unnecessary encapsulation.
- **English Only**: All code comments, documentation, commit messages, and descriptions must be in English.
- **No Style Tweaks**: PRs for formatting or beautification alone are rejected.
- **Line-by-Line Review**: All contributions are manually reviewed. AI-generated code must be thoroughly cleaned.
- **Functional Purity**: Keep code concise, clear, and minimal.

## Framework Patterns

- HuggingFace `PreTrainedModel` inheritance for model registry integration
- Custom `SpeechConnector` modules map speech VAE dimensions to LLM hidden states
- Diffusion head uses DPM-Solver scheduler for speech token generation
- Streaming processing uses 60-second segments for long audio (>600s)
- Special tokens: `<|speech_start|>`, `<|speech_end|>`, `<|speech_pad|>`

## Common Entry Points

- `VibeVoiceASRForConditionalGeneration` - Main ASR model class
- `VibeVoiceForConditionalGeneration` - Main TTS model class
- `VibeVoiceProcessor` / `VibeVoiceASRProcessor` - Input processing
- `AudioStreamer` / `AsyncAudioStreamer` - Streaming output handling
